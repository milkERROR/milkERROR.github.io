开题报告内容

研究的基本内容：

在家庭财务协同管理系统的设计与实现过程中，将深入研究系统各项功能模块，以满足现代家庭对财务管理系统的全面需求。家庭成员组织管理模块：实现基于角色的访问控制（即RBAC），区分家长、财务官、普通成员等角色。各角色权限等级不同，细分单项权限也可以灵活更改。允许用于添加、删除和修改家庭成员信息。实时协同记账模块：实现收入和支出信息录入、查询和统计等功能，收支项的自定义分类，包含日常消费、银行账户储蓄、投资、借贷等，支持多人同时操作，包括审批流程，实现大额或计划外支出的家庭内部管控。智能分摊模块：系统内置多种账单分摊算法（如AA、比例、指定金额等）。预算协同管理模块：支持制定家庭总预算、分配个人子预算，并实时监控执行。预算可以随时更改。家庭财务目标与储蓄计划管理：实现共同目标的创建、进度追踪与贡献度统计。实现收入和储蓄计划的录入、查询和统计。多维度数据可视化分析模块：集成图表库，提供个人支出比例图表、成员对比、趋势分析、健康度评分等报表，财务报告共享，家庭“共读、共议、共决策”。智能提醒模块：实现账单到期、预算超支等提醒。

允许用户为每笔账单打上隐私标签，控制可见范围，如“公共账目”“个人隐私账目”“夫妻账目”“仅家长可见”“自定义可见范围”等。课选择将个人敏感消费以匿名形式计入统计，不展示详细内容。

通过这些功能模块的设计和实现，最后构建一个功能全面、易于操作的家庭财务协同管理系统。

拟解决的主要问题：

在家庭财务协同管理系统的设计与实现过程中，将解决以下几个问题：多用户协同下的数据一致性和操作冲突，可通过WebSocket实时同步以及乐观锁策略解决。家庭场景下，灵活复杂的财务权限管控问题，可通过自定义RBAC模型，数据行级权限解决。自动化、公平化的账单分摊与结算问题，可通过规则引擎配置分摊算法解决。从杂乱数据到直观决策支持的转化，可通过数据聚合与ECharts可视化解决。系统安全性、可扩展性与可维护性问题，将通过SpringBoot分层架构的模块化设计解决。

开发流程（deep seek方案）

以下是开发家庭财务协同管理系统前端页面的详细流程，每一步都明确了需要完成的内容，以确保最终构建出功能全面、易于操作的系统。

1. 需求分析与功能确认
  • 内容：与产品经理、后端开发人员共同梳理各模块的具体需求，明确每个页面的核心功能。
    • 实时协同记账模块：收入/支出录入、自定义分类（日常消费、银行账户、储蓄、投资、借贷等）、多人协同操作（审批流程）、大额/计划外支出审批。
    • 智能分摊模块：AA分摊、比例分摊、指定金额分摊等算法。
    • 预算协同管理：家庭总预算、个人子预算制定与修改、实时执行监控。
    • 家庭财务目标与储蓄计划：目标创建、进度追踪、贡献度统计、储蓄计划录入与查询。
    • 多维度数据可视化：个人支出比例图表、成员对比、趋势分析、健康度评分、财务报告共享。
    • 智能提醒：账单到期提醒、预算超支提醒。
    • 隐私控制：账单隐私标签（公共、个人、夫妻、仅家长、自定义）、匿名统计。
  • 产出：功能列表、页面流程图、数据交互说明。
2. 技术选型与框架搭建
  • 内容：选择适合的前端技术栈。
    • 框架：Vue3 或 React（考虑生态和团队熟悉度），搭配 TypeScript 提供类型安全。
    • 状态管理：Pinia（Vue）或 Redux Toolkit（React），用于管理家庭共享数据（如账单、预算、目标）。
    • 路由：Vue Router 或 React Router，实现多页面导航。
    • UI组件库：Ant Design Vue/React 或 Element Plus，快速构建表单、表格、弹窗。
    • 数据可视化：ECharts 或 AntV，实现图表展示。
    • 实时通信：WebSocket 或 Socket.IO 用于协同操作通知。
    • 构建工具：Vite 或 Webpack。
  • 产出：项目脚手架、依赖安装、基础配置（ESLint、Prettier）。
3. UI/UX设计与原型制作
  • 内容：根据功能需求设计用户界面，制作高保真原型。
    • 设计登录/注册页、首页仪表盘（展示总览、健康度评分）。
    • 记账页面：收支录入表单、自定义分类选择器、隐私标签选择。
    • 账单列表：支持筛选、审批状态标识。
    • 预算管理页：进度条、编辑表单。
    • 目标页：创建目标卡片、贡献进度条。
    • 可视化报表页：图表布局、共享报告按钮。
    • 设置页：隐私偏好设置。
  • 产出：Figma/Sketch 设计稿、交互说明。
4. 项目初始化与目录结构规划
  • 内容：搭建规范的项目目录结构，便于团队协作。
    • 常见结构：src/api（接口请求）、src/assets（静态资源）、src/components（公共组件）、src/views（页面）、src/store（状态）、src/router、src/utils（工具函数）、src/types（类型定义）。
    • 创建基础布局组件（Layout），包含侧边栏、顶部导航。
  • 产出：可运行的初始项目，目录清晰。
5. 公共组件开发
  • 内容：开发可复用的基础组件，提高开发效率。
    • 通用组件：表单输入框、按钮、卡片、加载状态、空状态。
    • 业务组件：分类选择器、隐私标签选择器、金额输入框、审批按钮、进度条。
    • 图表封装：基于ECharts封装通用的图表组件（柱状图、饼图、折线图）。
  • 产出：组件库文档（Storybook 可选），各组件独立可用。
6. 状态管理与数据流设计
  • 内容：设计全局状态，模拟或对接后端数据模型。
    • 定义用户信息、家庭信息、账单列表、预算数据、目标数据等 store。
    • 实现协同操作的状态同步（例如 WebSocket 接收新账单后更新本地列表）。
    • 处理审批流程的状态变化（待审批、已通过、已拒绝）。
  • 产出：状态管理模块，支持实时更新。
7. 接口对接与数据处理
  • 内容：编写 API 请求模块，对接后端接口。
    • 使用 Axios 封装请求，统一处理错误、加载状态。
    • 定义所有接口路径和请求/响应数据类型（TypeScript）。
    • 实现登录鉴权（Token 管理）、权限控制（家长、成员等角色）。
    • 实现数据缓存策略（如预算数据、分类字典）。
  • 产出：API 服务层，可在页面中调用。
8. 路由与导航配置
  • 内容：配置前端路由，实现页面跳转和权限守卫。
    • 定义路由表：/dashboard、/bills、/budget、/goals、/reports、/settings 等。
    • 添加路由守卫：检查登录状态、角色权限（例如家长才能看到审批管理）。
    • 配置动态路由（可选，根据家庭角色显示不同菜单）。
  • 产出：完整的路由系统，菜单导航联动。
9. 功能模块实现（按顺序开发）
9.1 实时协同记账模块
  • 内容：
    • 开发账单列表页：展示所有账单，支持按时间、分类、隐私标签筛选；显示审批状态；支持点击详情。
    • 开发新增/编辑账单页：表单包含金额、分类、日期、描述、隐私标签选择；支持上传凭证（图片）。
    • 实现审批流程：家长或指定审批人可在账单详情页进行“通过”或“拒绝”操作；审批后状态更新。
    • 实现多人同时操作：通过 WebSocket 推送新账单或状态变更，页面实时刷新。
  • 产出：可录入、查询、审批账单，实时同步。
9.2 智能分摊模块
  • 内容：
    • 开发分摊设置界面：在新增支出时，可选择分摊方式（AA、比例、指定金额），并选择参与成员。
    • 前端根据选择实时计算各成员应承担金额。
    • 将分摊结果保存到账单中，后续在统计中体现。
  • 产出：支持多种分摊算法，计算结果准确。
9.3 预算协同管理模块
  • 内容：
    • 开发预算总览页：显示家庭总预算和成员子预算的进度条，可实时监控执行情况（支出 vs 预算）。
    • 开发预算编辑页：家长可设置/修改总预算，成员可申请调整个人子预算（需审批）。
    • 实时监控：当支出增加时，预算进度自动更新，并触发超支提醒。
  • 产出：预算制定与监控功能，支持动态调整。
9.4 家庭财务目标与储蓄计划管理
  • 内容：
    • 开发目标列表页：展示家庭共同目标（如旅游、购车）及其进度。
    • 开发创建目标页：输入目标名称、金额、截止日期，并设置贡献成员。
    • 开发储蓄计划录入：定期定额储蓄计划，自动计算贡献度。
    • 统计展示：显示每个成员对目标的贡献金额及占比。
  • 产出：目标创建、进度追踪、贡献统计。
9.5 多维度数据可视化分析
  • 内容：
    • 集成图表库，在报表页展示：
      • 个人支出比例图表（饼图/环形图）。
      • 成员支出对比（柱状图）。
      • 家庭支出趋势（折线图，按周/月）。
      • 财务健康度评分（仪表盘）。
    • 实现财务报告共享：生成报告页面，可分享给家庭成员或导出为图片/PDF。
    • 添加“共读、共议、共决策”功能：在报告下方添加评论或点赞区域。
  • 产出：丰富的图表和报告，支持互动。
9.6 智能提醒模块
  • 内容：
    • 实现提醒的触发与展示：在首页或顶部导航添加提醒图标，下拉显示待办提醒（账单到期、预算超支）。
    • 与后端配合，通过 WebSocket 或轮询获取新提醒。
    • 提醒可点击跳转到相关页面（如超支预算页、待支付账单）。
  • 产出：实时提醒功能。
9.7 隐私控制与权限管理
  • 内容：
    • 在账单录入时，提供隐私标签选择（公共、个人、仅家长、自定义）。
    • 在列表和详情页，根据当前用户角色和隐私标签控制可见性：例如个人隐私账目仅本人和家长可见；夫妻账目仅夫妻双方可见。
    • 实现匿名统计：当用户选择匿名时，在统计图表中只显示“匿名用户”分类，不展示具体描述。
  • 产出：细粒度的隐私控制，满足家庭隐私需求。
10. 集成测试与调试
  • 内容：
    • 单元测试：使用 Vitest 或 Jest 测试核心工具函数和组件。
    • 集成测试：模拟用户操作，测试主要流程（如创建账单、审批、查看报表）。
    • 跨浏览器测试：Chrome、Firefox、Safari 兼容性。
    • 调试协同冲突：模拟多人同时操作，确保状态一致。
  • 产出：测试报告，修复发现的 Bug。
11. 性能优化
  • 内容：
    • 代码分割：按路由动态加载页面组件。
    • 虚拟列表：如果账单列表过长，使用虚拟滚动。
    • 图表按需加载：ECharts 仅引入使用的图表类型。
    • 缓存优化：使用 IndexedDB 缓存常用数据（如分类字典）。
    • 图片优化：上传的凭证压缩，使用 WebP 格式。
  • 产出：性能评分提升，首屏加载时间缩短。
12. 部署与上线
  • 内容：
    • 配置生产环境变量（API 地址等）。
    • 构建项目：npm run build，生成 dist 文件夹。
    • 部署至服务器或云服务（如 Nginx、OSS 静态托管）。
    • 配置 HTTPS 和 CDN 加速。
    • 上线前进行最后回归测试。
  • 产出：可访问的线上系统。
13. 用户反馈与迭代
  • 内容：
    • 收集用户（家庭成员）使用反馈，记录新需求。
    • 根据反馈优化交互体验，修复线上 Bug。
    • 定期发布新版本，增加功能（如更多分摊算法、更丰富的图表）。
  • 产出：持续迭代的计划和版本更新。

协同功能

要测试家庭财务系统的 协同功能 ，最直观的方法是模拟“多成员参与”和“实时交互”。您可以按照以下步骤进行操作：

1. 准备多个测试账号

协同的核心是多用户。您可以打开两个不同的浏览器（例如 Chrome 和 Edge），或者一个普通窗口加一个隐身窗口：

  • 浏览器 A (家长) ：登录账号 A（例如 admin ）。
  • 浏览器 B (成员) ：注册并登录账号 B（例如 member1 ）。

2. 构建家庭协同关系

目前系统在注册时会自动创建独立的默认家庭。为了测试协同，您需要手动将账号 B 加入账号 A 的家庭：

  1. 获取账号 B 的 ID ：在数据库 user 表中查看账号 B 的 id 。
  2. 在浏览器 A 中添加成员 ：
    • 进入“成员管理” -> “成员列表”。
    • 点击“添加成员”，输入账号 B 的用户名或 ID。
    • 分配角色（如 MEMBER ）。
    • （注：如果 UI 还没实现搜索添加，您可以直接在数据库 family_member 表中插入一条数据，让 A 和 B 的 family_id 相同） 。

3. 测试关键协同场景

场景一：账单实时通知 (WebSocket)

  • 操作 ：浏览器 A 进入首页，浏览器 B 进入“收支录入”页面。

  • 动作 ：浏览器 B 提交一笔新账单。

  • 验证 ：浏览器 A 的首页应该会立即弹出右下角的通知（由 home.html 中的 WebSocket 实现），提示有新动态。
    场景二：费用分摊 (Expense Sharing)

  • 操作 ：浏览器 B 进入“收支录入”。

  • 动作 ：录入一笔“聚餐”费用，在“分摊模式”中选择 AA 分摊 ，并勾选账号 A 和账号 B。

  • 验证 ：

    • 浏览器 B 提交后，在“账单列表”中查看。
    • 浏览器 A 刷新“账单列表”，应该能看到这笔由 B 录入但包含 A 分摊份额的账单。
      场景三：预算预警 (Budget Monitoring)

  • 操作 ：浏览器 A 在“预算管理”中设定本月总预算为 1000 元。

  • 动作 ：浏览器 B 录入一笔 1200 元的大额支出。

  • 验证 ：

    • 系统会自动检测到预算超支。
    • A 和 B 的首页“预算进度”卡片都会变为警告色（由 home.html 逻辑控制）。
    • 点击“系统通知”，应该能看到一条“预算超支”的提醒。
      场景四：隐私权限隔离

  • 操作 ：浏览器 B 录入一笔账单，在“隐私标签”中选择 “仅自己可见” 。

  • 动作 ：浏览器 A 查看“账单列表”。

  • 验证 ：浏览器 A 应该 无法 看到这笔账单，确保了家庭成员间的财务隐私。

项目理解（cursor auto提供）

一、项目是做什么的

这是一个 理财/金融管理 Web 系统，可以理解为一个小型“理财平台”：

  • 普通用户：登录后可以买零钱理财、定期理财、基金、工资理财，申请网贷，绑银行卡，看自己的理财记录和资金流水等。

  • 管理员：管理各类理财产品、审核网贷、管理用户和权限、发新闻、管银行信息等。

技术上是一个 Spring Boot + MyBatis + 页面模板 的 Java Web 项目，用 Shiro 做登录和权限控制。

二、整体架构（分层）

整体是常见的 “控制层 → 业务层 → 数据访问层” 结构，再加实体和公共组件：

同时还有：

  • entity：和数据库表对应的 Java 类（User、Admin、Loan、ChangeMoney 等）。

  • common：通用东西（统一返回格式 Msg、登录拦截、Shiro 配置、编码过滤器等）。

所以：架构 = 经典三层 + 实体 + 公共组件。

三、包结构大致分工
包名 作用（通俗说）
controller 处理每个“页面/接口”：谁点哪个链接、点哪个按钮，由哪个 Controller 方法管。例如登录、注册、我的理财、网贷、银行卡等。
service 业务逻辑：比如“买零钱理财”要查产品、创建用户理财记录、记流水，这些写在 Service 里。
service.impl 上面 Service 接口的具体实现类，里面会调用 Mapper。
mapper 和数据库打交道的接口，每个接口对应一张表或一类 SQL（查用户、查理财产品、插流水等）。
entity 数据库表对应的 Java 类，以及一些 Example（用于条件查询）。
common 公共配置和工具：统一返回格式、登录校验、Shiro 权限、错误页、编码过滤等。

启动类 FinanceApplication 里做了三件事：扫 Mapper、开事务、扫 Servlet/Filter，所以整个应用从这一处启动。

四、主要功能在代码里是怎么体现的
  1. 登录与权限

  • LoginController：用户/管理员登录校验、注册；用 Shiro 的 Subject.login(token) 做认证，成功后根据身份跳转 /user/index.html 或 /admin/index.html。

  • ShiroConfig：配置哪些路径必须登录、哪些放行（如 /、/login/、静态资源），以及 /user/ 需要 user 角色、/admin/** 需要 admin 角色。

  • LoginHandlerInterceptor：从 Session 里看有没有 loginUser 或 loginAdmin，没有就拦下来跳到登录页（当前在 MyMvcConfig 里被注释掉了，实际主要靠 Shiro 在做访问控制）。

  1. 用户端理财相关

  • 零钱理财：ChangeMoneyController → 展示零钱理财产品列表；用户“买入”时在 Service 里创建 UserChangeMoney 记录，并插入一条 FlowOfFunds 流水。

  • 我的理财汇总：MyFinanceController 的“我的理财”页面会查当前用户持有的：零钱理财、工资理财、定期理财、基金，并支持“撤销”某笔理财（把状态改为 3）。

  • 网贷：LoanController → 用户申请网贷（金额、期限、利率）、查看“我的借贷”列表；管理员侧有审核等（在别的 Controller 里）。

  • 其他类似：定期理财（TermFinancial）、基金（FundProduct）、工资理财（PayMoney）都是“产品表 + 用户持有表 + 流水”的模式，Controller 负责页面和接口，Service 负责业务，Mapper 负责数据库。

  1. 通用数据与返回格式

  • 列表分页用 PageHelper（如 LoanController 里 PageHelper.startPage + PageInfo）。

  • 前后端交互的 JSON 用 Msg 封装：Msg.success() / Msg.fail()，需要带数据时用 add(“key”, value)，前端根据 code（如 100 成功、200 失败）和 extend 里的数据做事。

  1. 视图与静态资源

  • 页面用 Thymeleaf 模板，放在 resources 下；MyMvcConfig 里把 / 和 /index.html 直接映射到登录页，/toregister.html 映射到注册页。

  • 静态资源（如 /asserts/、/js/ 等）在 Shiro 里做了放行，不登录也能访问。

五、数据流举例（买一笔零钱理财）

  1. 用户在前端选产品、点“买入”，请求发到：/user/buyChangeMoney（POST）。

  2. ChangeMoneyController.buyChangeMoney() 收到 changeMoneyId、userId。

  3. Controller 调 Service：查产品 ChangeMoney → 建一条 UserChangeMoney（用户持有）→ 再插一条 FlowOfFunds（资金流水）。

  4. Service 内部通过各个 Mapper（如 ChangeMoneyMapper、UserChangeMoneyMapper、FlowOfFundsMapper）读写数据库。

  5. 返回 Msg.success() 或 Msg.fail() 给前端，前端再刷新或跳转。

其它“买入”类功能（定期、基金、工资理财等）流程类似，只是表和业务字段不同。

六、小结（一句话版）

  • 架构：Spring Boot + 三层（Controller → Service → Mapper）+ 实体 + common（Msg、Shiro、拦截器等）。

  • 功能：用户端做各类理财购买、持有、撤销和网贷申请；管理员端做产品、借贷、用户与权限管理；登录与权限主要由 Shiro 和 Session 配合完成。

  • 代码分工：Controller 管“接到什么请求、返回什么”；Service 管“这件事怎么做”；Mapper 管“怎么读库写库”；entity 是表对应的模型；common 是全局共用配置和工具。
